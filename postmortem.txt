ROOT CAUSE ANALYSIS

IMMEDIATE CAUSE: Human console change

Developer opened sg-0abc123 port 22 to 0.0.0.0/0 via AWS Console for debugging
access. This created the security exposure (CIS 5.2 violation).

Context: Legitimate debugging workflow, intended as temporary change, planned
to revert after session. No malicious intent. No policy violation (Console
access is permitted for authorized engineers).

ROOT CAUSE OF 19-HOUR DELAY: System design failure

The security exposure itself was human-caused. The 19-hour delay in fixing it
was system-caused.

Detection was fast (4 minutes). Remediation was slow (19 hours). The gap was
not due to engineer negligence or slow response times. It was the inevitable
result of architectural decisions in our detection and correlation systems.

The question is not "Why did the developer open the port?"
The question is "Why did it take 19 hours to fix what we detected in 4 minutes?"

CONTRIBUTING FACTORS TO DELAY:

1. SILOED DETECTION SYSTEMS
   - Terraform polling: Detects infrastructure drift
   - CSPM scanning: Detects security posture violations
   - VPC flow logs: Detects network anomalies

   Each system operates independently with separate alerting pipelines.
   No shared context. No automatic correlation.

2. CONTEXT LOSS ACROSS TOOLS
   - Terraform sees: "ingress.cidr changed from 10.0.0.0/8 to 0.0.0.0/0"
   - CSPM sees: "CIS 5.2 violation on sg-0abc123"
   - VPC flow sees: "Anomalous SSH attempts on i-0abc123"

   Same root cause, three separate observations, zero connection.

3. TICKETING SYSTEM ISOLATION
   - INFRA-2847: Created by Jenkins, assigned to InfraOps queue
   - SEC-9103: Created by Security Hub, assigned to SecOps queue
   - NET-4521: Created by VPC flow analyzer, assigned to NetOps queue

   No cross-referencing. No shared incident ID. No composite view.

4. DETECTION LATENCY VARIANCE
   - Terraform: 4-minute polling (fast, but context-free)
   - CSPM: 3-hour batch scan (delayed security visibility)
   - VPC flow: Daily aggregation (slowest detection path)

   First detection was infrastructure-only. Security detection came 3.6
   hours later. By then, InfraOps had already closed their ticket.

5. INSUFFICIENT OPERATIONAL CONTEXT
   - InfraOps ticket showed state diff only (no security implication)
   - SecOps ticket showed compliance violation only (no change history)
   - NetOps ticket showed traffic anomaly only (no infrastructure link)

   No single team had complete picture to assess criticality correctly.

WHAT DIDN'T FAIL:
  - Developer: Followed legitimate debugging workflow, intended to revert
  - InfraOps engineer: Followed SLA, assessed available data correctly
  - SecOps engineer: Triaged HIGH priority alert per standard procedure
  - NetOps engineer: Identified anomaly during routine review
  - Detection systems: All functioned as designed (Terraform, CSPM, VPC flow)

WHAT DID FAIL:
  - System architecture: Designed to produce isolated, uncorrelated alerts
  - Correlation layer: Does not exist (manual discovery only)
  - Shared context: No mechanism to connect infrastructure + security + network
  - Drift attribution: No automated root cause analysis across domains

WHY THIS MATTERS

This incident class represents a systemic gap in modern cloud operations:

  "We can detect infrastructure drift in minutes,
   but it takes hours to understand what it means."

The InfraOps engineer wasn't wrong.
The SecOps engineer wasn't wrong.
The system was designed to produce this outcome.

Three teams, three tickets, three separate investigations of the SAME event.
Detection is fast. Correlation is manual. Context is lost.

The operational question is not "Who missed this?"
The operational question is "Why did the system require manual correlation?"

LESSONS LEARNED

1. Detection speed is not the bottleneck. Correlation is.

   We detected the change in 4 minutes (Terraform polling). We detected the
   security violation in 3.6 hours (CSPM scan). But correlation took 9+ hours
   because it required manual cross-team investigation.

2. Context matters more than velocity.

   Fast detection without operational context produces low-priority tickets
   that get closed prematurely. InfraOps saw a CIDR change. SecOps saw a CIS
   violation. NetOps saw traffic anomaly. Nobody saw the full story.

3. Siloed tools create siloed incidents.

   Terraform, Security Hub, and VPC flow logs are excellent tools. But they
   operate independently, producing separate alerts with no shared incident ID.
   The correlation layer does not exist.

4. This is a design problem, not a people problem.

   Blaming the InfraOps engineer misses the point. They followed procedure,
   assessed the data they had, and made a reasonable decision. The system
   didn't give them the information needed to assess criticality correctly.

5. The generation problem is solved. The operations problem is not.

   We can generate infrastructure code perfectly (IaC, AI-assisted). We can
   deploy it reliably (CI/CD, GitOps). But we cannot yet operate it with the
   same level of automation and intelligence. Day 1 is solved. Day 2+ is not.
