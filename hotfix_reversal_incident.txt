EXECUTIVE SUMMARY
-----------------
Production API service experienced two distinct outage periods caused by
database connection timeouts. Initial incident was resolved via emergency
Console hotfix at 14:14 UTC. Automated drift remediation system detected
the Console change as unauthorized drift and reverted the fix at 14:19 UTC,
causing service to degrade again. This extended the P1 incident by 22
additional minutes.

Root cause: Drift detection system lacked operational context awareness.
System did not check for active incidents before executing remediation.

KEY FINDING: Technically correct automation + operationally blind = P1 extension

IMPACT ANALYSIS
---------------

Service Degradation Periods:
  Period 1: 14:02 - 14:14 (12 minutes) - Original incident
  Period 2: 14:23 - 14:33 (10 minutes) - Automation-induced recurrence

Total Customer Impact: 22 minutes of degraded service

Without automation interference: 12 minutes (Period 1 only)
With automation interference: 22 minutes (both periods)
Additional impact caused by automation: 10 minutes (83% increase)

Human Time Impact:
  - Incident Commander: 24 additional minutes investigating
  - On-call engineer paged twice (confusion, stress)
  - Re-investigation from scratch (duplicated effort)

Trust Impact:
  - Incident Commander quote: "I can't trust the automation if it's
    going to undo my emergency fixes during a P1."
  - Team consensus: Disable drift remediation until this is fixed


ROOT CAUSE ANALYSIS
-------------------

IMMEDIATE CAUSE:
Drift detection system auto-remediated a security group change that was
actually an emergency hotfix, not unauthorized drift.

CONTRIBUTING FACTORS:
1. Drift detection system had no integration with incident management
   (PagerDuty). It could not distinguish between:
     - Unauthorized drift (should remediate)
     - Emergency hotfix during active incident (should NOT remediate)

2. Risk score algorithm (0.55) was below auto-merge threshold (0.6),
   qualifying the change for automatic remediation without human review.

3. No suppression window for Console changes. Best practice: suppress
   drift alerts for 2 hours after Console modification to allow humans
   time to create follow-up PR.

4. No "active_incident" field in DriftEvent schema to carry operational
   context from PagerDuty into drift detection decisions.

ROOT CAUSE (System Design Failure):
The drift remediation system was designed with a TECHNICAL CORRECTNESS
model but lacked OPERATIONAL CONTEXT AWARENESS. It correctly identified
that actual state != desired state (Terraform), but it did not consider
WHY the divergence occurred or WHEN remediation was appropriate.

A technically correct system that is operationally unaware will make
correct decisions at exactly the wrong time.


LESSONS LEARNED
---------------

LESSON 1: Operational Context > Technical Correctness
Why it matters: A system can be 100% technically correct and still
cause incidents if it lacks operational awareness. During a P1, humans
are making emergency decisions. Automation must respect that context.

LESSON 2: Integration is Not Optional
Why it matters: The drift system existed in isolation from incident
management (PagerDuty). This created a dangerous blind spot. Systems
that make autonomous decisions MUST integrate with operational state.

LESSON 3: Conservative Thresholds Save Incidents
Why it matters: If auto-merge threshold had been 0.3 instead of 0.6,
this change (risk 0.55) would have required human review. The human
would have seen it was their own emergency fix and rejected the PR.

LESSON 4: Speed is Not Always the Goal
Why it matters: Auto-merge in 8 minutes sounds great in demos. But
during a P1, that speed turned into a liability. Sometimes slow is safe.


CORRECTIVE ACTIONS (Completed)

[COMPLETED] 2024-03-16
Action: Implement PagerDuty -> EventBridge -> DriftEvent integration
Owner: platform-team
Details:
  - PagerDuty webhook triggers EventBridge rule on incident.triggered
  - Lambda function sets active_incident=true in DynamoDB state table
  - All DriftEvent objects check state table and populate active_incident field
  - When active_incident=true, ALL tier1/tier2 auto-merge is blocked
Testing: Validated with synthetic incident, drift detection correctly paused

[COMPLETED] 2024-03-17
Action: Add active_incident field to DriftEvent schema
Owner: platform-team
File: code/core/schemas.py (line 188-195)
Details:
  - New boolean field: active_incident (default: false)
  - Populated from PagerDuty integration state
  - Passed through Detective -> Surgeon -> Checker -> OPA pipeline
  - Acts as global circuit breaker for all automation

[COMPLETED] 2024-03-17
Action: Update OPA policies with active_incident guard
Owner: platform-team
Files:
  - code/policies/tier1.rego (line 56): added active_incident check
  - code/policies/tier2.rego (line 68-72): added BLOCK rule for active incidents
  - code/policies/tier3.rego (line 72-76): added hard block with override path
Details:
  - If active_incident=true, ALL auto-remediation blocked
  - Override path: incident commander can add "incident/override" label
  - Override requires explicit justification in PR comment

[COMPLETED] 2024-03-18
Action: Lower auto-merge threshold from 0.6 to 0.3
Owner: platform-team
Rationale:
  - Analysis of 90 days of drift events:
      40% of events: risk < 0.3 (safe for auto-merge)
      35% of events: risk 0.3-0.6 (borderline, require review)
      25% of events: risk > 0.6 (definitely require review)
  - New threshold: Only truly low-risk changes auto-merge
  - Trade-off: More human reviews, but fewer automation mistakes

[COMPLETED] 2024-03-19
Action: Implement Console change suppression window (2 hours)
Owner: platform-team
Details:
  - CloudTrail event source_ip detection: if NOT from known CIDR ranges,
    assume Console change (human at keyboard)
  - Suppress drift alerts for resource_id for 2 hours after Console change
  - Gives human time to create follow-up PR without automation interference
  - After 2 hours, drift detection resumes normally

[COMPLETED] 2024-03-20
Action: Add "incident/override" label and workflow
Owner: platform-team
Details:
  - New GitHub label: "incident/override"
  - Requires incident commander role (PagerDuty on-call)
  - Must include justification comment on PR
  - Allows remediation during active incident when genuinely needed (rare)
  - All overrides logged to audit trail

APPENDIX A: TECHNICAL DETAILS
------------------------------

Drift Detection Configuration (at time of incident):
  Version: drift-detection-service v1.1.4
  Polling interval: 5 minutes
  Risk threshold (auto-merge): < 0.6
  Environment: production (us-east-1)
  Terraform backend: S3 (terraform-state-prod)
  State locking: DynamoDB (terraform-locks)
  PagerDuty integration: NOT IMPLEMENTED (added post-incident)

Risk Score Calculation (for sg-0abc123 change):
  Base risk: 0.4 (security group = medium criticality)
  Blast radius multiplier: 1.2 (affects 12 EC2 instances, 1 RDS)
  Environment multiplier: 1.1 (production)
  Policy violation multiplier: 1.0 (no violations detected)
  Final risk score: 0.4 * 1.2 * 1.1 * 1.0 = 0.528 (rounded to 0.55)
  Decision: 0.55 < 0.6 (auto-merge threshold) -> AUTO-MERGE APPROVED

Why Risk Score Was Too Low:
  - Algorithm did not consider "Console change during business hours" signal
  - Did not consider "change by on-call engineer" signal
  - Did not consider temporal proximity to alerts (5 minutes after alert)
  - All of these signals indicate emergency hotfix, not drift
